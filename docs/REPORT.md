## 1. Назначение и область применения

DAST Analyzer представляет собой программный комплекс, предназначенный для автоматизированного обнаружения уязвимостей в веб-приложениях методом динамического тестирования безопасности (Dynamic Application Security Testing, DAST).

В отличие от статического анализа (SAST), динамический метод не требует доступа к исходному коду: тестирование осуществляется непосредственно в отношении работающего приложения путём формирования реальных HTTP-запросов, инъекции тестовых нагрузок и последующего анализа ответов сервера. Данный подход позволяет выявлять уязвимости, проявляющиеся только в условиях реального исполнения, в том числе обусловленные ошибками конфигурации промежуточного программного обеспечения.

**Целевая аудитория:** специалисты по информационной безопасности, разработчики программного обеспечения, системные администраторы и аудиторы защищённости, осуществляющие проверку подконтрольных им веб-ресурсов.

**Ограничения применения.** Система предназначена исключительно для тестирования ресурсов, в отношении которых пользователь располагает соответствующими полномочиями. Сканирование сторонних ресурсов без явного разрешения их владельца является нарушением действующего законодательства.

## 2. Функциональные возможности системы

### 2.1 Аутентификация и управление доступом

Система реализует многоуровневую модель аутентификации:

- регистрация и вход по паре электронная почта / пароль;
- двухфакторная аутентификация (2FA) на основе одноразовых паролей по стандарту TOTP (RFC 6238);
- аутентификация через протокол OAuth 2.0 с использованием стороннего провайдера (GitHub);
- разграничение ролей: пользователь (`user`) и администратор (`admin`);
- управление сессиями с возможностью их принудительного завершения;
- выдача и отзыв долгосрочных API-токенов для программного доступа.

### 2.2 Сканирование веб-приложений

Основная функциональность системы включает:

- автоматический обход (краулинг) целевого ресурса с обнаружением страниц, GET-параметров, HTML-форм и JavaScript-маршрутов;
- анализ HTTP-заголовков ответа на соответствие требованиям безопасности;
- инъекцию тестовых нагрузок (payloads) для выявления следующих классов уязвимостей:

| Класс уязвимости | Идентификатор CWE |
|---|---|
| SQL-инъекция (SQLi) | CWE-89 |
| Межсайтовый скриптинг (XSS) | CWE-79 |
| Подделка запросов на стороне сервера (SSRF) | CWE-918 |
| Открытый редирект | CWE-601 |
| Инъекция заголовков | CWE-113 |
| Ошибки конфигурации безопасности | CWE-16 |

- управление ходом сканирования в режиме реального времени: постановка на паузу, возобновление, принудительная отмена;
- ведение детального журнала событий по каждой задаче сканирования.

### 2.3 Отчётность

По завершении сканирования формируется структурированный отчёт, содержащий:

- перечень обнаруженных уязвимостей с классификацией по критичности (Critical, High, Medium, Low, Info);
- ссылки на описания уязвимостей в базах MITRE CWE и OWASP Top 10 2021;
- готовые `curl`-команды для воспроизведения каждого обнаруженного дефекта безопасности;
- рекомендации по устранению уязвимостей.

### 2.4 Управление словарями нагрузок

- загрузка пользовательских текстовых файлов-словарей; поддерживается объём до 10 ГБ;
- автоматическое подключение всех словарей пользователя при запуске сканирования;
- удаление ранее загруженных словарей.

### 2.5 Административные функции

- просмотр и управление учётными записями: активация, деактивация;
- просмотр и отзыв API-токенов всех пользователей;
- просмотр и принудительное завершение активных сессий.

## 3. Архитектура системы

### 3.1 Общая схема

Система построена по принципу разделения компонентов: клиентская часть (SPA), серверный API, воркер (обработчик задач сканирования), реляционная СУБД и брокер очередей функционируют как независимые сервисы, взаимодействующие по чётко определённым интерфейсам.

```
Клиент (браузер)
       │ HTTPS
       ▼
 ┌───────────┐
 │   Nginx   │  — обратный прокси, раздача статики SPA, завершение TLS
 └─────┬─────┘
       │ HTTP (внутренняя сеть Docker)
       ▼
 ┌─────────────┐
 │ FastAPI API │  — REST API, бизнес-логика, аутентификация
 └──────┬──────┘
        │                          │
        ▼                          ▼
 ┌────────────┐           ┌──────────────┐
 │ PostgreSQL │           │ Redis (очередь│
 │   (данные) │           │   scan_queue) │
 └────────────┘           └──────┬───────┘
                                 │ blpop
                                 ▼
                          ┌────────────┐
                          │   Worker   │  — краулер, движок атак
                          └────────────┘
```

### 3.2 Описание компонентов

| Компонент | Роль | Технология |
|---|---|---|
| **Frontend** | Одностраничное веб-приложение (SPA) | React 18, TypeScript, Vite |
| **Backend API** | REST API, бизнес-логика, аутентификация | Python 3.11, FastAPI, SQLAlchemy (async) |
| **Worker** | Асинхронный обработчик задач сканирования | Python 3.11, asyncio, httpx |
| **PostgreSQL** | Основное реляционное хранилище | PostgreSQL 16 |
| **Redis** | Очередь задач и брокер событий | Redis 7 |
| **Nginx** | Обратный прокси, раздача статики, TLS | Nginx 1.25 |
| **Certbot** | Получение и обновление TLS-сертификатов | Let's Encrypt / certbot |

### 3.3 Взаимодействие компонентов

1. Браузер обращается к Nginx по протоколу HTTPS.
2. Nginx перенаправляет запросы с префиксом `/api/*` на внутренний сервис Backend; остальные запросы обслуживаются статически (файлы SPA).
3. Backend выполняет аутентификацию, валидацию входных данных и бизнес-логику.
4. При создании задачи сканирования Backend публикует идентификатор задачи в очередь Redis посредством команды `RPUSH scan_queue <scan_id>`.
5. Worker читает очередь Redis командой `BLPOP scan_queue` с таймаутом 1 секунда и запускает обработку очередной задачи в виде асинхронной сопрограммы.
6. Результаты (обнаруженные уязвимости, статус, журнал) сохраняются в PostgreSQL через ORM SQLAlchemy.
7. Клиент периодически опрашивает Backend для получения актуального статуса задачи и результатов.

### 3.4 Изоляция сети

Все внутренние сервисы (PostgreSQL, Redis, Backend, Worker) функционируют в изолированной Docker-сети `dast_internal` и не доступны из внешней сети напрямую. Единственным публично доступным сервисом является Nginx (порты 80 и 443). Тестовый стенд DVWA вынесен в отдельную сеть `dvwa_net`.

## 4. Принцип работы механизма сканирования

### 4.1 Жизненный цикл задачи сканирования

Задача сканирования в процессе своего существования последовательно проходит следующие состояния:

```
[pending] → [running] → [finished]
                     ↘ [paused]  → [pending] → [running] ...
                     ↘ [cancelled]
                     ↘ [failed]
```

- **pending** — задача создана и помещена в очередь Redis; воркер ещё не приступил к обработке.
- **running** — воркер приступил к выполнению задачи.
- **paused** — выполнение приостановлено по запросу пользователя; воркер завершил текущую итерацию атаки и остановился.
- **finished** — сканирование завершено в штатном режиме.
- **cancelled** — выполнение прекращено по запросу пользователя; частичные результаты сохранены.
- **failed** — выполнение прервано вследствие неустранимой ошибки или превышения лимита времени.

### 4.2 Механизм управления состоянием (пауза и возобновление)

Управление состоянием реализовано через совместное использование базы данных и асинхронного события (`asyncio.Event`):

**Пауза:**
1. API записывает статус `paused` в БД.
2. Воркер в фоновом режиме (функция `check_pause_signals`) опрашивает БД для всех активных задач. При обнаружении статуса `paused` вызывается метод `orchestrator.stop()`, который устанавливает флаг `_stop_event`.
3. Краулер и движок атак проверяют флаг `_stop_event` на границах итераций и корректно завершают работу.
4. Оркестратор фиксирует в БД накопленные результаты (уязвимости) и завершает выполнение. Статус в БД не перезаписывается — он уже установлен на шаге 1.

**Возобновление:**
1. API проверяет, что текущий статус задачи равен `paused`, устанавливает статус `pending` и публикует идентификатор задачи в очередь Redis.
2. Воркер подхватывает идентификатор из очереди, создаёт новый экземпляр `ScanOrchestrator` и запускает сканирование заново (с фазы краулинга).

Архитектурное решение — разделение ответственности: API является единственным источником записи статусов `paused` и `cancelled` в БД; оркестратор записывает только статусы `running`, `finished` и `failed`. Данный подход исключает состояние гонки, при котором оркестратор (завершающий обработку предыдущей итерации) мог бы перезаписать статус `pending`, установленный запросом на возобновление.

### 4.3 Фаза 1 — Краулинг (обход ресурса)

Краулинг реализован как обход в ширину (BFS, Breadth-First Search). Каждая страница посещается однократно; повторные визиты исключаются через множество посещённых URL (`visited_urls`).

Параметр **глубина (max_depth)** определяет максимальное число переходов по ссылкам от стартового URL:

| Глубина | Что охватывается |
|---|---|
| 0 | Только стартовая страница |
| 1 | Стартовая страница и все ссылки с неё |
| 2 | Дополнительно — все ссылки со страниц первого уровня |
| N | Все страницы в N переходах от стартовой |

Краулер извлекает из каждой страницы:
- все гиперссылки (`<a href=...>`);
- параметры GET-запросов (из обнаруженных URL);
- HTML-формы с их полями и адресами обработки (`action`);
- JavaScript-маршруты (статический анализ содержимого тегов `<script>`).

Краулер поддерживает конфигурацию аутентификации для исследования защищённых разделов сайта:

| Тип | Механизм |
|---|---|
| `form` | Автоматическое заполнение и отправка формы входа |
| `cookie` | Передача готового значения Cookie |
| `bearer` | Передача Bearer-токена в заголовке Authorization |
| `basic` | HTTP Basic Authentication |

Для корректного завершения краулинг ограничен по времени: выделяется 60 % от общего таймаута задачи.

### 4.4 Фаза 1.5 — Анализ заголовков безопасности

Для каждого из первых 50 обнаруженных URL выполняется отдельный HTTP-запрос. Анализатор `SecurityHeadersAnalyzer` проверяет наличие следующих заголовков безопасности в ответе:

- `Content-Security-Policy`
- `X-Frame-Options`
- `X-Content-Type-Options`
- `Strict-Transport-Security`
- `Referrer-Policy`
- `Permissions-Policy`

Отсутствие каждого из перечисленных заголовков фиксируется как уязвимость с соответствующим уровнем критичности и рекомендацией по устранению.

### 4.5 Фаза 2 — Атака (инъекция нагрузок)

**Движок нагрузок (`PayloadEngine`)** формирует список целей для атаки из результатов краулинга:

- **GET-цели**: для каждого параметра каждого URL генерируется набор тестовых запросов, в которых значение параметра заменяется на payload. Максимальное число GET-целей ограничено 800.
- **POST-цели**: для каждого поля каждой HTML-формы генерируется набор тестовых данных с подставленным payload. Максимальное число POST-целей ограничено 400.

Нагрузки (payloads) загружаются из встроенных текстовых файлов-словарей (`sqli.txt`, `xss.txt`, `ssrf.txt`, `open_redirect.txt`, `header_injection.txt`) и из пользовательских словарей, загруженных через интерфейс системы.

**Алгоритм атаки** для каждой цели:

1. Выполняется «базовый» запрос (baseline) — запрос с нейтральным значением параметра. Фиксируются: HTTP-статус, размер тела ответа, время ответа.
2. Выполняется тестовый запрос с подставленным payload.
3. Анализаторы сравнивают тестовый ответ с базовым и выносят вердикт:

**Сигнатурный анализатор (`SignatureAnalyzer`)** ищет в теле ответа признаки успешной эксплуатации уязвимости: фрагменты SQL-ошибок, отражение переданного XSS-payload, перенаправление на целевой адрес SSRF и т.д.

**Эвристический анализатор (`HeuristicAnalyzer`)** анализирует изменения параметров ответа относительно baseline: изменение кода статуса, значительное изменение размера тела, аномальное увеличение времени ответа (признак time-based инъекции).

При обнаружении уязвимости создаётся запись в таблице `Vulnerability`, содержащая тип, URL, параметр, использованный payload, уровень критичности, идентификаторы CWE и OWASP, доказательства (evidence) и рекомендацию по устранению.

## 5. Модель данных

### 5.1 Сущности и атрибуты

**User** — учётная запись пользователя:

| Атрибут | Тип | Описание |
|---|---|---|
| `id` | UUID7 | Первичный ключ |
| `email` | String | Адрес электронной почты (уникальный) |

| `hashed_password` | String | Хэш пароля (bcrypt, cost 12) |
| `role` | Enum | Роль: `user` или `admin` |
| `is_active` | Boolean | Активность учётной записи |
| `totp_secret` | String | Секрет TOTP для 2FA (nullable) |

| Атрибут | Тип | Описание |
|---|---|---|
| `id` | UUID7 | Первичный ключ |
| `user_id` | FK → User | Владелец сессии |
| `refresh_token_hash` | String | SHA-256 хэш refresh-токена |
| `user_agent` | String | User-Agent браузера |
| `ip_address` | String | IP-адрес клиента |
| `expires_at` | DateTime | Срок действия |

**APIToken** — API-токен для программного доступа:

| Атрибут | Тип | Описание |
|---|---|---|
| `id` | UUID7 | Первичный ключ |
| `owner_id` | FK → User | Владелец токена |
| `name` | String | Произвольное название |
| `token_hash` | String | SHA-256 хэш значения токена |
| `last_used_at` | DateTime | Время последнего использования |

**Scan** — задача сканирования:

| Атрибут | Тип | Описание |
|---|---|---|
| `id` | UUID7 | Первичный ключ |
| `owner_id` | FK → User | Владелец задачи |
| `target_url` | String | URL целевого ресурса |
| `status` | Enum | Текущее состояние задачи |
| `max_depth` | Integer | Максимальная глубина краулинга |
| `timeout_seconds` | Integer | Таймаут выполнения (секунды) |
| `config` | JSONB | Параметры аутентификации, статистика краулинга |
| `started_at` | DateTime | Время начала выполнения |
| `finished_at` | DateTime | Время завершения |

**Vulnerability** — обнаруженная уязвимость:

| Атрибут | Тип | Описание |
|---|---|---|
| `id` | UUID7 | Первичный ключ |
| `scan_id` | FK → Scan | Принадлежность сканированию |
| `vuln_type` | Enum | Тип уязвимости (sqli, xss и т.д.) |
| `severity` | Enum | Критичность (critical / high / medium / low / info) |
| `url` | String | URL, на котором обнаружена уязвимость |
| `parameter` | String | Уязвимый параметр |
| `method` | String | HTTP-метод (GET / POST) |
| `payload` | String | Использованная нагрузка |
| `evidence` | JSONB | Доказательства: фрагмент ответа, curl-команда, статус |
| `recommendation` | String | Рекомендация по устранению |

**Wordlist** — пользовательский словарь нагрузок:

| Атрибут | Тип | Описание |
|---|---|---|
| `id` | UUID7 | Первичный ключ |
| `owner_id` | FK → User | Владелец словаря |
| `name` | String | Отображаемое имя |
| `file_path` | String | Путь к файлу на сервере |
| `file_size` | Integer | Размер файла в байтах |
| `is_builtin` | Boolean | Признак встроенного словаря |

### 5.2 Диаграмма связей

```
User ──< UserSession
User ──< APIToken
User ──< Scan ──< Vulnerability
User ──< Wordlist
```

### 5.3 Идентификаторы UUID7

В качестве первичных ключей всех сущностей используются идентификаторы формата UUID версии 7 (UUIDv7). В отличие от UUID4, UUIDv7 монотонно возрастает во времени, что обеспечивает эффективную индексацию в СУБД и позволяет восстанавливать хронологию создания объектов без дополнительных полей сортировки.

## 6. Программный интерфейс (REST API)

### 6.1 Общие принципы

API системы реализован в соответствии с принципами архитектурного стиля REST. Все запросы и ответы используют формат JSON. Интерактивная документация API доступна по адресам `/api/docs` (Swagger UI) и `/api/redoc`.

### 6.2 Аутентификация запросов

Все защищённые эндпоинты требуют передачи валидного токена в HTTP-заголовке:

```
Authorization: Bearer <access_token>
```

Запросы без валидного токена или с истёкшим токеном отклоняются с кодом ответа **401 Unauthorized**.

### 6.3 Основные группы эндпоинтов

| Группа | Префикс | Назначение |
|---|---|---|
| Аутентификация | `/api/auth` | Регистрация, вход, 2FA, OAuth, обновление токена |
| Пользователи | `/api/users` | Профиль, смена пароля, аватар, API-токены |
| Сканирования | `/api/scans` | CRUD задач сканирования, управление состоянием |
| Отчёты | `/api/scans/{id}/report` | Перечень уязвимостей по задаче |
| Журнал событий | `/api/scans/{id}/logs` | Поток событий сканирования |
| Словари | `/api/wordlists` | Загрузка, перечень, удаление |
| Администрирование | `/api/admin` | Управление пользователями, сессиями, токенами |
| Служебный | `/health` | Проверка работоспособности |

### 6.4 Коды ответа

| Код | Условие |
|---|---|
| 200 | Запрос выполнен успешно |
| 201 | Ресурс создан |
| 400 | Некорректный запрос (ошибка валидации) |
| 401 | Отсутствует или недействителен токен аутентификации |
| 403 | Доступ запрещён (недостаточно прав) |
| 404 | Ресурс не найден |
| 413 | Размер загружаемого файла превышен |
| 415 | Неподдерживаемый тип файла |
| 429 | Превышен лимит запросов (Rate Limit) |
| 500 | Внутренняя ошибка сервера |

## 8. Руководство пользователя

### 8.1 Требования к рабочему месту

Для работы с системой требуется веб-браузер одной из следующих версий или более поздних: Google Chrome 90, Mozilla Firefox 88, Apple Safari 14, Microsoft Edge 90. Установка специализированного программного обеспечения на рабочем месте не требуется.

### 8.2 Регистрация и вход

**Регистрация новой учётной записи:**

1. Открыть стартовую страницу приложения.
2. Нажать кнопку «Создать аккаунт».
3. Заполнить форму: адрес электронной почты, уникальное имя пользователя (3–64 символа, латиница и цифры), пароль (не менее 8 символов, буквы и цифры).
4. Нажать «Зарегистрироваться».

**Вход по логину и паролю:** ввести адрес электронной почты и пароль, нажать «Войти».

**Вход через GitHub OAuth:** нажать «Войти через GitHub», в открывшемся окне GitHub подтвердить предоставление доступа.

**Двухфакторная аутентификация (при включённой 2FA):** после ввода логина и пароля ввести 6-значный код из приложения-аутентификатора (Google Authenticator, Authy или аналогичного). Код действителен в течение 30 секунд.

### 8.3 Создание задачи сканирования

1. На главной странице нажать «Новое сканирование».
2. Указать URL целевого ресурса (полный адрес, включая схему).
3. Опционально выбрать пользовательские словари нагрузок.
4. Нажать «Запустить».

После создания задача переходит в статус «Ожидание» (`pending`) и помещается в очередь обработки. При наличии свободного воркера обработка начнётся немедленно, и статус изменится на «Выполняется» (`running`).

### 8.4 Мониторинг хода сканирования

На странице задачи отображается текущий статус и журнал событий в режиме реального времени. Сканирование проходит три последовательных фазы:

| Фаза | Описание |
|---|---|
| Краулинг | BFS-обход страниц целевого ресурса, сбор форм и параметров |
| Анализ заголовков | Проверка HTTP-заголовков ответа |
| Атака | Инъекция нагрузок, анализ ответов, регистрация уязвимостей |

**Управление в процессе выполнения:**

- **Пауза** — приостанавливает выполнение по завершении текущей итерации атаки; результаты сохраняются.
- **Возобновить** — повторно помещает задачу в очередь и перезапускает сканирование с фазы краулинга.
- **Отмена** — прекращает выполнение; частичные результаты сохраняются и доступны в отчёте.

### 8.5 Просмотр отчёта

По завершении сканирования формируется таблица обнаруженных уязвимостей. Доступна фильтрация по уровню критичности. Нажатие на строку таблицы раскрывает детальную информацию:

- использованный payload;
- `curl`-команда для воспроизведения (с кнопкой копирования);
- фрагмент тела HTTP-ответа;
- идентификаторы CWE и OWASP;
- рекомендация по устранению.

### 8.6 Управление словарями

Раздел «Словари» предназначен для загрузки пользовательских текстовых файлов-словарей. Каждая строка файла является одной нагрузкой; строки, начинающиеся с символа `#`, считаются комментариями. Кодировка файла: UTF-8; максимальный размер: 10 ГБ. Все загруженные словари автоматически подключаются при запуске каждого последующего сканирования.

### 8.7 Управление профилем и API-токенами

В разделе «Профиль» доступны: изменение имени пользователя и пароля, загрузка аватара (JPEG / PNG, до 5 МБ), включение и отключение 2FA.

API-токены предназначены для программного доступа к системе без интерактивной аутентификации. Значение токена отображается однократно при создании и в дальнейшем недоступно. Токен передаётся в заголовке `Authorization: Bearer <значение>`.

### 8.8 Административная панель

Раздел «Администратор» доступен только пользователям с ролью `admin`. Функциональность включает:

- **Пользователи** — просмотр всех учётных записей, активация и деактивация.
- **API-токены** — просмотр и отзыв токенов всех пользователей.
- **Сессии** — просмотр и принудительное завершение активных сессий (отображаются IP-адрес, User-Agent, время создания).

## 10. Меры обеспечения информационной безопасности

### 10.1 Политика единого источника (SOP)

Браузер применяет политику SOP, блокируя кросс-доменные запросы к API системы от сторонних источников. Приложение не ослабляет данную политику и не использует конфигурации типа `Access-Control-Allow-Origin: *`.

### 10.2 Политика безопасности содержимого (CSP)

Nginx возвращает заголовок `Content-Security-Policy` при каждом ответе:

```
default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline';
img-src 'self' data:; font-src 'self';
```

Директивы ограничивают загрузку ресурсов только origin приложения, что предотвращает выполнение инъецированных сценариев при XSS-атаках.

### 10.3 Политика CORS

Заголовок `Access-Control-Allow-Origin` возвращается исключительно для доменов из переменной окружения `ALLOWED_ORIGINS`. Реализовано через middleware `CORSMiddleware` FastAPI. Запросы с источником, не входящим в список, не получают CORS-заголовков и блокируются браузером.

### 10.4 Защита от CSRF

Система использует архитектуру SPA с аутентификацией посредством JWT Bearer-токенов. Токены передаются в заголовке `Authorization`, который браузер не добавляет автоматически к кросс-доменным запросам. Данный механизм исключает вектор CSRF без применения дополнительных CSRF-токенов.

### 10.5 Аутентификация API (Bearer-токены)

Два типа токенов:

| Тип | Срок действия | Назначение |
|---|---|---|
| `access_token` | 60 минут | Авторизация API-запросов |
| `refresh_token` | 30 суток | Получение нового access-токена |

Токены подписываются алгоритмом HS256. Refresh-токены и API-токены хранятся в БД в виде SHA-256 хэша.

### 10.6 Ограничение частоты запросов (Rate Limiting)

| Эндпоинт | Лимит |
|---|---|
| `POST /api/auth/register` | 5 запросов/мин с IP |
| `POST /api/auth/login` | 10 запросов/мин с IP |
| `POST /api/auth/2fa/verify` | 10 запросов/мин с IP |
| Прочие эндпоинты | 60 запросов/мин с IP |

При превышении лимита возвращается ответ **429 Too Many Requests**.

### 10.7 Сквозное журналирование запросов (Request ID)

Каждому входящему запросу присваивается уникальный идентификатор в формате UUIDv7. Идентификатор фиксируется в журнале и возвращается клиенту в заголовке `X-Request-Id`. Это обеспечивает трассировку запросов при расследовании инцидентов.

### 10.8 Hardening сервера эксплуатации

**SSH (`/etc/ssh/sshd_config`):**
```
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
MaxAuthTries 3
Port 2222
```

**Межсетевой экран (UFW):**
```bash
ufw default deny incoming
ufw allow 2222/tcp  # SSH
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable
```

**Защита от перебора (fail2ban):**
```ini
[sshd]
enabled = true
port    = 2222
maxretry = 5
bantime  = 3600
```

### 10.9 TLS-конфигурация Nginx

```nginx
ssl_protocols             TLSv1.2 TLSv1.3;
ssl_ciphers               ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:...;
ssl_prefer_server_ciphers off;
ssl_session_cache         shared:SSL:10m;
```

Поддерживаются только TLS 1.2 и TLS 1.3. Устаревшие протоколы (SSLv3, TLS 1.0, TLS 1.1) отключены. Заголовок `server_tokens off` скрывает версию Nginx.

**Заголовки безопасности Nginx:**

| Заголовок | Значение | Назначение |
|---|---|---|
| `Strict-Transport-Security` | `max-age=63072000; includeSubDomains` | Принудительное HTTPS (HSTS) |
| `X-Content-Type-Options` | `nosniff` | Запрет MIME-sniffing |
| `X-Frame-Options` | `DENY` | Защита от Clickjacking |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Ограничение передачи Referer |

### 10.10 Безопасность контейнеров

Контейнеры запускаются от имени непривилегированного пользователя `appuser` (без прав root). Запись в постоянное хранилище осуществляется только через именованные тома Docker. Контейнеры не используют флаг `--privileged`.

### 10.11 Хэширование паролей

Пароли пользователей хранятся исключительно в виде хэша, вычисленного алгоритмом bcrypt с коэффициентом сложности 12. Исходные значения паролей нигде не сохраняются. Refresh-токены и API-токены хранятся в виде SHA-256 хэша.

### 10.12 Двухфакторная аутентификация (2FA/TOTP)

Реализована в соответствии со стандартом RFC 6238 посредством библиотеки `pyotp`. Процесс аутентификации при включённой 2FA:

1. Проверка логина и пароля — выдаётся краткосрочный `pre_auth_token` (5 минут).
2. Проверка 6-значного TOTP-кода — при успехе выдаются access и refresh токены.

## 11. Процесс непрерывной интеграции и доставки (CI/CD)

### 11.1 Инструменты и сервисы

| Инструмент | Роль |
|---|---|
| GitHub Actions | Автоматизация pipeline |
| GitHub Container Registry (GHCR) | Реестр Docker-образов |
| SonarCloud | Статический анализ кода (SAST) |
| Aqua Trivy | Сканирование уязвимостей в образах |

### 11.2 Pipeline при отправке изменений (push / pull request)

При каждой отправке изменений в ветку `main` или открытии pull request выполняются следующие этапы:

1. **Тестирование backend** — запуск набора тестов pytest с базой данных SQLite в памяти.
2. **Проверка типов frontend** — сборка TypeScript-проекта (`tsc --noEmit`).
3. **Статический анализ кода (SAST)** — отправка кода в SonarCloud; анализ Python и TypeScript.
4. **Сканирование Docker-образа** — сборка образа backend и его проверка инструментом Aqua Trivy на наличие известных CVE. Отчёт сохраняется как артефакт pipeline.
5. **Развёртывание на сервере** — подключение по SSH и выполнение команд: `git pull`, `docker compose build`, `docker compose up -d`.

### 11.3 Семантическое версионирование и публикация релизов

Релизы оформляются посредством Git-тегов в формате `vX.Y.Z` согласно методике семантического версионирования (semver.org):

- **MAJOR** (X) — несовместимые изменения API.
- **MINOR** (Y) — новая функциональность без нарушения обратной совместимости.
- **PATCH** (Z) — исправление дефектов.

При создании тега формата `v*.*.*` pipeline дополнительно:

- выполняет аутентификацию в GHCR;
- публикует Docker-образы backend и nginx с тегом версии и тегом `latest`;
- создаёт GitHub Release с автоматически сгенерированными примечаниями к выпуску.

```bash
# Пример создания нового релиза
git tag v1.2.0
git push origin v1.2.0
```

### 11.4 Хранение секретов CI/CD

Все чувствительные данные (SSH-ключи, токены, пароли) хранятся в GitHub Actions Secrets. В журналах выполнения pipeline значения секретов автоматически маскируются. Секреты не передаются в pull request из форков репозитория.

Перечень необходимых секретов:

| Секрет | Описание |
