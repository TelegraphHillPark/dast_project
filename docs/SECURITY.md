# Описание мер обеспечения безопасности — DAST Analyzer

## 1. Общие положения

Настоящий документ описывает технические меры обеспечения информационной безопасности, реализованные в системе DAST Analyzer. Меры охватывают следующие уровни: безопасность интерфейсов взаимодействия, безопасность сервера эксплуатации, транспортную безопасность, безопасность хранения данных и процессы непрерывной интеграции.

---

## 2. Безопасность интерфейсов взаимодействия

### 2.1 Политика единого источника (Same-Origin Policy, SOP)

Браузер применяет политику SOP, блокируя кросс-доменные запросы от сторонних источников к API системы. Приложение не ослабляет данную политику намеренно и не использует небезопасные конфигурации типа `Access-Control-Allow-Origin: *`.

### 2.2 Политика безопасности содержимого (Content Security Policy, CSP)

Nginx возвращает заголовок `Content-Security-Policy` при каждом ответе:

```
Content-Security-Policy:
  default-src 'self';
  script-src  'self' 'unsafe-inline';
  style-src   'self' 'unsafe-inline';
  img-src     'self' data:;
  font-src    'self';
```

Директивы `default-src`, `script-src` и `style-src` ограничивают источники загрузки ресурсов origin приложения, что предотвращает выполнение инъецированных сценариев при XSS-атаках.

### 2.3 Политика совместного использования ресурсов (CORS)

Сервер возвращает заголовок `Access-Control-Allow-Origin` исключительно для доменов, перечисленных в переменной окружения `ALLOWED_ORIGINS`. Обработка CORS реализована в FastAPI-приложении:

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.cors_origins,  # список доверенных источников
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

Запросы с происхождением, не входящим в список, не получают CORS-заголовков. Браузер блокирует такие ответы в соответствии с политикой SOP.

### 2.4 Аутентификация API (Bearer-токены)

Доступ ко всем защищённым эндпоинтам требует передачи валидного JWT-токена в заголовке HTTP-запроса:

```
Authorization: Bearer <access_token>
```

Запросы, не содержащие валидного токена или содержащие токен с истёкшим сроком действия, отклоняются с кодом ответа **401 Unauthorized**.

Система использует два типа токенов:

| Тип | Время действия | Назначение |
|---|---|---|
| `access_token` | 60 минут | Авторизация API-запросов |
| `refresh_token` | 30 суток | Получение нового access-токена |

Токены подписываются алгоритмом **HS256** с секретным ключом из переменной окружения `SECRET_KEY`.

### 2.5 Защита от CSRF

Система построена на архитектуре SPA (Single Page Application) с аутентификацией посредством JWT Bearer-токенов. Токены передаются исключительно в заголовке `Authorization`, который браузер не добавляет автоматически к кросс-доменным запросам. Данный механизм устраняет вектор атаки CSRF без необходимости использования дополнительных CSRF-токенов.

### 2.6 Ограничение частоты запросов (Rate Limiting)

Защита от DoS-атак и атак перебора реализована посредством библиотеки `slowapi`. Лимиты устанавливаются на уровне отдельных эндпоинтов:

| Эндпоинт | Лимит |
|---|---|
| `POST /api/auth/register` | 5 запросов/минуту с IP |
| `POST /api/auth/login` | 10 запросов/минуту с IP |
| `POST /api/auth/2fa/verify` | 10 запросов/минуту с IP |
| Прочие эндпоинты | 60 запросов/минуту с IP |

При превышении установленного лимита сервер возвращает ответ с кодом **429 Too Many Requests**.

### 2.7 Сквозное журналирование запросов (Request ID)

Каждый входящий HTTP-запрос обрабатывается middleware, которое:

1. Генерирует уникальный идентификатор запроса в формате **UUIDv7**.
2. Записывает в журнал метод, URL, код ответа, время обработки и идентификатор запроса.
3. Возвращает идентификатор клиенту в заголовке ответа `X-Request-Id`.

Пример записи в журнале:

```
INFO  [2024-01-15 10:23:45] POST /api/auth/login 200 0.045s req_id=01HQ...
```

## 3. Hardening сервера эксплуатации

### 3.1 Настройка безопасного SSH-доступа

Конфигурационный файл SSH-сервера (`/etc/ssh/sshd_config`) должен содержать следующие директивы:

```
Port 2222                    # нестандартный порт (вместо 22)
PermitRootLogin no           # запрет входа под пользователем root
PasswordAuthentication no    # запрет аутентификации по паролю
PubkeyAuthentication yes     # аутентификация по открытому ключу
MaxAuthTries 3               # максимальное число попыток аутентификации
```

Применение изменений:

```bash
systemctl restart sshd
```

### 3.2 Политика сложных паролей

Установка модуля парольной политики:

```bash
apt install libpam-pwquality
```

Конфигурационный файл `/etc/security/pwquality.conf`:

```
minlen = 12
dcredit = -1
ucredit = -1
lcredit = -1
ocredit = -1
```

### 3.3 Защита от перебора паролей (fail2ban)

```bash
apt install fail2ban
```

Конфигурационный файл `/etc/fail2ban/jail.local`:

```ini
[sshd]
enabled  = true
port     = 2222
maxretry = 5
bantime  = 3600
findtime = 600
```

Применение:

```bash
systemctl enable --now fail2ban
```

### 3.4 Настройка межсетевого экрана (UFW)

```bash
ufw default deny incoming
ufw default allow outgoing
ufw allow 2222/tcp    # SSH (нестандартный порт)
ufw allow 80/tcp      # HTTP
ufw allow 443/tcp     # HTTPS
ufw allow 8888/tcp    # DVWA (тестовый стенд)
ufw enable
ufw status verbose
```

## 4. Доступ к приложению через Nginx с защитой TLS

### 4.1 Роль Nginx

Nginx функционирует в режиме обратного прокси (reverse proxy):

- принимает входящие HTTPS-соединения на порту 443;
- завершает TLS-сессии;
- проксирует запросы к URI `/api/*` на внутренний сервис backend;
- раздаёт статические файлы фронтенда (SPA);
- перенаправляет HTTP-запросы (порт 80) на HTTPS (код 301).

### 4.2 TLS-сертификат

TLS-сертификат выпускается центром сертификации Let's Encrypt посредством утилиты certbot с методом подтверждения владения доменом `webroot`. Сертификат обновляется автоматически при наличии менее 30 суток до истечения срока действия (задание cron, выполняемое ежедневно в 03:00).

При отсутствии сертификата Let's Encrypt (например, при использовании IP-адреса без домена) Nginx автоматически генерирует самоподписанный сертификат на срок 10 лет.

### 4.3 Параметры TLS-конфигурации

```nginx
ssl_protocols             TLSv1.2 TLSv1.3;
ssl_ciphers               ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:
                          ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
ssl_prefer_server_ciphers off;
ssl_session_cache         shared:SSL:10m;
ssl_session_timeout       1d;
```

Поддерживаются протоколы TLS 1.2 и TLS 1.3. Устаревшие протоколы SSLv3, TLS 1.0 и TLS 1.1 отключены. Применяются только стойкие шифры на основе ECDHE.

### 4.4 Заголовки безопасности

Nginx добавляет следующие заголовки безопасности к каждому ответу:

| Заголовок | Значение | Назначение |
|---|---|---|
| `Strict-Transport-Security` | `max-age=63072000; includeSubDomains` | Принудительное использование HTTPS (HSTS) |
| `X-Content-Type-Options` | `nosniff` | Запрет определения MIME-типа по содержимому |
| `X-Frame-Options` | `DENY` | Запрет встраивания страниц в фреймы (защита от Clickjacking) |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Ограничение передачи Referer |
| `Content-Security-Policy` | см. раздел 2.2 | Ограничение источников ресурсов |

Заголовок `server_tokens off` скрывает версию Nginx в ответах сервера.

## 5. Безопасность контейнеров

### 5.1 Запуск от непривилегированного пользователя

В `Dockerfile` приложение запускается от имени специально созданного непривилегированного пользователя:

```dockerfile
RUN useradd --no-create-home --shell /bin/false appuser
USER appuser
```

Контейнеры не запускаются с флагом `--privileged` и не имеют расширенных прав.

### 5.2 Изоляция файловой системы

Запись в постоянное хранилище допускается исключительно через именованные тома Docker (`volumes`): `uploads_data` для загружаемых файлов и `wordlists_data` для словарей. Корневая файловая система контейнера не монтируется на хост.

## 6. Безопасность хранения данных

### 6.1 Хэширование паролей

Пароли пользователей хранятся в базе данных исключительно в виде хэша, вычисленного алгоритмом **bcrypt** (фактор сложности 12). Исходные значения паролей нигде не сохраняются.

### 6.2 Хранение токенов

Refresh-токены и API-токены хранятся в базе данных в виде **SHA-256 хэша**. В случае компрометации базы данных злоумышленник не может использовать хранимые значения для аутентификации, поскольку исходные токены недоступны.

### 6.3 Секреты и переменные окружения

Все чувствительные параметры конфигурации (секретный ключ, пароли, токены OAuth) хранятся в переменных окружения и передаются в контейнеры через файл `.env`. Файл `.env` добавлен в `.gitignore` и не включается в репозиторий исходного кода.

## 7. Безопасность процесса CI/CD

### 7.1 Хранение секретов

Все чувствительные данные, используемые в pipeline (SSH-ключи, пароли, токены), хранятся в **GitHub Actions Secrets**. В журналах выполнения pipeline значения секретов автоматически маскируются символами `***`.

### 7.2 Статический анализ кода (SAST)

При каждом запуске pipeline выполняется статический анализ исходного кода инструментом **SonarCloud**. Анализ охватывает код на Python (backend) и TypeScript (frontend), выявляет потенциальные уязвимости, дефекты и нарушения стандартов написания кода.

### 7.3 Сканирование Docker-образов

Docker-образ backend сканируется инструментом **Aqua Trivy** на наличие известных уязвимостей (CVE) в системных пакетах и зависимостях Python. Отчёт о результатах сканирования сохраняется как артефакт pipeline.

### 7.4 Версионирование дистрибутивов

Релизы системы оформляются посредством Git-тегов в формате `vX.Y.Z` (семантическое версионирование). При создании тега pipeline автоматически:

- публикует Docker-образы в GitHub Container Registry (GHCR) с соответствующим тегом версии и тегом `latest`;
- создаёт GitHub Release с автоматически сгенерированными примечаниями к выпуску.

## 8. Двухфакторная аутентификация (2FA)

Система поддерживает двухфакторную аутентификацию на основе одноразовых паролей (TOTP) в соответствии со стандартом RFC 6238. Реализация выполнена посредством библиотеки `pyotp`. При включённой 2FA процесс аутентификации включает два этапа:

1. Проверка логина и пароля — выдаётся кратковременный `pre_auth_token` (5 минут).
2. Проверка 6-значного кода TOTP — при успехе выдаются полноценные access и refresh токены.
