## 1. Аутентификация и авторизация

### JWT-токены

Система использует два вида токенов:

- **Access-токен** — JWT с алгоритмом HS256, живёт **60 минут**. Подписывается ключом из `SECRET_KEY`. Передаётся в заголовке `Authorization: Bearer`.
- **Refresh-токен** — случайная строка, живёт **30 дней**. В базе данных хранится только SHA-256-хеш, не сам токен. При обновлении access-токена текущий refresh инвалидируется и выдаётся новый (token rotation).

Если refresh-токен скомпрометирован — злоумышленник может обновить access-токен один раз, но не более: после первого использования прежний refresh уже недействителен. Все активные сессии пользователя видны в профиле и могут быть завершены вручную.

### API-ключи

Используются для доступа без браузера. Генерируются как случайная строка с префиксом `dast_`. В БД хранится только SHA-256-хеш ключа — если база данных утечёт, реальные ключи восстановить невозможно. Сам ключ показывается пользователю **один раз** при создании.

Передаются в заголовке `X-API-Key`. FastAPI проверяет каждый запрос: либо валидный JWT, либо валидный API-ключ.

### Двухфакторная аутентификация (2FA)

TOTP по RFC 6238 (временные одноразовые коды, период 30 секунд). Реализация через библиотеку PyOTP. Секрет шифруется симметрично перед записью в базу данных — в случае утечки БД коды из него не восстановить без ключа шифрования.

При входе с включённой 2FA система сначала проверяет пароль и возвращает `pre_auth_token` (JWT с коротким TTL). Затем пользователь вводит TOTP-код, и только после его проверки выдаётся полноценная сессия. Это предотвращает перебор TOTP без знания пароля.

### OAuth (GitHub)

Реализован через библиотеку Authlib по протоколу OAuth 2.0. Состояние (state) проверяется при callback — защита от CSRF в OAuth-потоке. Email от GitHub считается верифицированным.

### Ролевая модель

Две роли: `user` и `admin`. Admin-эндпоинты (`/api/admin/*`) проверяются отдельной зависимостью `require_admin` — она проверяет не только то, что токен валидный, но и что у пользователя именно роль `admin`. Ошибочный доступ возвращает 403, не 401, чтобы не раскрывать существование защищённого ресурса.

Назначить первого администратора можно только через прямой доступ к БД — через API это невозможно (защита от privilege escalation через сам API).

## 2. Защита транспортного уровня

Nginx принимает все соединения на 80 и 443. HTTP-запросы (порт 80) перенаправляются на HTTPS через redirect 301.

TLS-конфигурация:
- Минимальная версия: **TLSv1.2**
- Поддерживаемые: TLSv1.2, TLSv1.3
- Шифры: `ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384`

По умолчанию используется самоподписанный сертификат (для разработки и демо). Для продакшена — заменяется на сертификат Let's Encrypt через Certbot (см. [DEPLOY.md](DEPLOY.md)).

Бэкенд не доступен напрямую снаружи — взаимодействует с Nginx только через внутреннюю Docker-сеть `dast_internal`. Порт 8000 проброшен для отладки, в продакшене его нужно закрыть (убрать `ports: "8000:8000"` из `docker-compose.yml`).

## 3. HTTP-заголовки безопасности

Заголовки выставляются в конфигурации Nginx:

| Заголовок | Значение | Назначение |
|---|---|---|
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | Запрет HTTP-соединений на 1 год |
| `Content-Security-Policy` | `default-src 'self'; script-src 'self' 'unsafe-inline'` | Ограничение источников ресурсов |
| `X-Frame-Options` | `DENY` | Защита от clickjacking |
| `X-Content-Type-Options` | `nosniff` | Запрет MIME-sniffing |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Контроль заголовка Referer |

`unsafe-inline` в CSP нужен для React — в текущей версии инлайн-стили используются для тем оформления. В будущем следует заменить на nonce-based CSP.

### CORS

Список разрешённых источников задаётся через переменную окружения `ALLOWED_ORIGINS`. В продакшене это должен быть конкретный домен, а не `*`. По умолчанию разрешён только `http://localhost`.

## 4. Ограничение запросов

Реализовано через библиотеку SlowAPI (обёртка над `limits`). Ключ идентификации — IP-адрес клиента.

| Эндпоинт | Лимит | Зачем |
|---|---|---|
| `POST /auth/register` | 5 / минута | Предотвращение массовой регистрации |
| `POST /auth/login` | 10 / минута | Замедление перебора паролей |
| `POST /auth/2fa/verify` | 10 / минута | Замедление перебора TOTP |
| `POST /scans` | 10 / минута | Предотвращение spam-сканирования |

На остальные эндпоинты глобальный лимит не применяется — это сделано намеренно, поскольку фронтенд опрашивает статус и логи каждые 2 секунды во время скана, и глобальный лимит вытеснял бы пользовательские действия.

При превышении лимита возвращается HTTP 429 с заголовком `Retry-After`.

> В продакшене рекомендуется дополнительно настроить `limit_req_zone` на уровне Nginx — это эффективнее, так как отсекает запросы до того, как они дойдут до Python.

## 5. Хранение чувствительных данных

### Пароли

Хешируются с помощью bcrypt через библиотеку `passlib`. Bcrypt включает соль автоматически, что исключает атаки по радужным таблицам.

### Токены и ключи

- Refresh-токены: только SHA-256-хеш в `user_sessions.token_hash`
- API-ключи: только SHA-256-хеш в `api_tokens.token_hash`
- TOTP-секреты: симметричное шифрование перед записью

### Секреты приложения

Все секреты передаются через переменные окружения (файл `.env`). Файл `.env` добавлен в `.gitignore` и не попадает в репозиторий.

`SECRET_KEY` — должен быть минимум 32 случайных байта. Если ключ скомпрометирован — все JWT немедленно теряют силу после смены ключа и перезапуска backend.

### Файлы пользователей (аватары)

Хранятся в Docker volume `uploads_data`, смонтированном в `/app/uploads/avatars/`. При загрузке проверяется MIME-тип файла. Файлы доступны по пути `/api/uploads/avatars/` через StaticFiles — без возможности обхода директории (`..`), так как FastAPI/Starlette нормализует пути.

Контейнер запускается от пользователя `appuser` (не root). Права на папку `uploads` выставляются при старте контейнера через `entrypoint.sh` с временным повышением до root через `gosu`.

## 6. Изоляция контейнеров

Все контейнеры находятся в одной Docker-сети `dast_internal` с драйвером `bridge`. Снаружи открыты только порты 80 и 443 (Nginx). PostgreSQL и Redis **не доступны** с хост-машины — только из контейнеров.

Бэкенд и воркер используют один Docker-образ, но запускаются разными командами. Воркер не открывает никаких портов.

Redis защищён паролем (`requirepass`). PostgreSQL — паролем пользователя.

## 7. Валидация входных данных

Все входящие данные проходят через Pydantic-схемы. FastAPI автоматически возвращает 422 при несоответствии типов или нарушении ограничений (min_length, max_length, ge, le).

Примеры ограничений:
- `target_url`: min_length=1, max_length=2048
- `max_depth`: ge=1, le=10
- `timeout_seconds`: ge=60, le=86400

Поле `target_url` валидируется браузером через `type="url"`, но это клиентская валидация — на сервере пока нет дополнительной проверки формата URL. Для внутренних адресов (`127.0.0.1`, `192.168.x.x`, `10.x.x.x`) ограничений нет — SSRF-сканер может намеренно обращаться к внутренним ресурсам как часть своей работы.

Загрузка файлов (аватары): проверяется Content-Type и максимальный размер 5 МБ. Файл сохраняется с UUID-именем, без сохранения оригинального имени.

## 8. Журналирование и трассировка

Каждый HTTP-запрос к API получает уникальный `X-Request-ID` на основе UUIDv7 — монотонно возрастающий, что облегчает сортировку логов по времени без обращения к таймстампам.

Middleware `RequestLoggingMiddleware` логирует:
- Метод и путь запроса
- HTTP-статус ответа
- Время выполнения (мс)
- Request ID

Логи выводятся в stdout контейнера в формате:
```
2026-05-19T12:00:00 INFO  dast.middleware POST /api/scans 201 142ms [req-id: 0195abc...]
```

Логи доступны через `docker compose logs`. В продакшене рекомендуется направлять их в централизованную систему (ELK, Loki и т.п.).

## 9. Работа сканера

### Правовые и этические ограничения

Сканер предназначен только для проверки систем, на которые у вас есть явное письменное разрешение. Несанкционированное сканирование чужих систем является нарушением законодательства.

Разрешённые цели для тестирования:
- Собственные приложения
- Специально уязвимые стенды (DVWA, WebGoat, Metasploitable)
- Сайты с явно указанной программой Bug Bounty, разрешающей автоматическое сканирование

### Что делает сканер

Сканер реально отправляет атакующие нагрузки на целевой сервер:
- SQL-инъекции в GET-параметры и POST-поля форм
- XSS-нагрузки с попыткой отражения
- SSRF-нагрузки с указанием внешних адресов
- Заголовки с нагрузками (Header Injection)

Это не пассивное наблюдение — запросы попадают в логи целевого сервера и могут срабатывать WAF/IDS.

### Изоляция от атак

Сканер атакует только целевой URL, указанный пользователем, и остаётся строго в пределах одного домена (нет кросс-доменных переходов). Пути из списка `excluded_paths` не посещаются и не атакуются — используйте его для защиты деструктивных операций (logout, смена пароля, удаление данных).

## 10. Известные ограничения

- **CSP `unsafe-inline`** — допускает инлайн-скрипты, что ослабляет защиту от XSS. Планируется замена на nonce-based CSP.
- **Порт 8000 открыт наружу** — в текущем `docker-compose.yml` порт бэкенда проброшен для удобства разработки. В продакшене нужно убрать строку `ports: "8000:8000"` из секции `backend`.
- **Самоподписанный сертификат** — в продакшене заменить на Let's Encrypt.
- **Нет CAPTCHA** на форме регистрации — только rate limiting 5/мин.
- **Отсутствует проверка владения доменом** — пользователь может запустить скан против любого URL. Нет технических препятствий для нацеливания на чужие сайты (только правовые).
- **Redis без TLS** — внутренняя сеть Docker шифрование не обеспечивает. Для высококритичных развёртываний следует настроить TLS для Redis.
