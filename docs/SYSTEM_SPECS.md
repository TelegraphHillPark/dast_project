# Описание системы DAST Analyzer

## 1. Общие сведения о системе

**Наименование системы:** DAST Analyzer — система динамического анализа безопасности веб-приложений.

**Назначение системы:** Автоматизированное обнаружение уязвимостей в веб-приложениях методом динамического тестирования (Dynamic Application Security Testing, DAST). Система осуществляет краулинг целевого ресурса, инъекцию тестовых полезных нагрузок в параметры запросов и формы, анализ ответов сервера и формирование структурированных отчётов об обнаруженных уязвимостях.

**Категория пользователей:** специалисты по информационной безопасности, разработчики программного обеспечения, системные администраторы.

## 2. Функциональные возможности системы

### 2.1 Аутентификация и управление доступом

- Регистрация и аутентификация пользователей по паре логин/пароль.
- Двухфакторная аутентификация (2FA) на основе одноразовых паролей (TOTP/OTP).
- Аутентификация через протокол OAuth 2.0 с использованием стороннего провайдера (GitHub).
- Разграничение ролей: пользователь (`user`) и администратор (`admin`).
- Управление сессиями пользователей с возможностью принудительного завершения.
- Управление API-токенами для доступа к внешнему HTTP API.

### 2.2 Функции сканирования

- Краулинг целевого веб-ресурса: обнаружение страниц, ссылок, форм и GET-параметров.
- Анализ заголовков HTTP-ответов на соответствие требованиям безопасности.
- Инъекция тестовых пейлоадов для выявления следующих классов уязвимостей:
  - SQL-инъекция (SQLi, CWE-89)
  - Межсайтовый скриптинг (XSS, CWE-79)
  - Подделка запросов на стороне сервера (SSRF, CWE-918)
  - Открытый редирект (CWE-601)
  - Инъекция заголовков (CWE-113)
  - Ошибки конфигурации безопасности (CWE-16)
- Управление сканированием в режиме реального времени: постановка на паузу, возобновление, отмена.
- Журналирование событий сканирования с привязкой к идентификатору запуска.

### 2.3 Отчётность

- Формирование структурированного отчёта по каждому сканированию.
- Классификация уязвимостей по критичности (Critical, High, Medium, Low, Info).
- Отображение идентификаторов CWE с гиперссылками на базу MITRE.
- Отображение идентификаторов OWASP Top 10 2021.
- Генерация curl-команд для воспроизведения каждой уязвимости.
- Фильтрация результатов по критичности.

### 2.4 Управление словарями пейлоадов

- Загрузка пользовательских файлов словарей (формат: текстовый файл, одна строка — один пейлоад).
- Поддержка файлов объёмом до 10 ГБ.
- Автоматическое использование пользовательских словарей при запуске сканирования владельца.
- Удаление словарей.

### 2.5 Управление профилем

- Изменение учётных данных (имя пользователя, пароль).
- Загрузка аватара пользователя.
- Включение и отключение двухфакторной аутентификации.

### 2.6 Административная панель

- Просмотр и управление учётными записями пользователей (активация, деактивация).
- Просмотр и отзыв API-токенов.
- Просмотр и завершение активных сессий.

## 3. Архитектура системы

### 3.1 Общая схема

Система построена по микросервисной архитектуре с разделением на следующие компоненты:

```
Клиент (браузер)
       │ HTTPS
       ▼
   Nginx (reverse proxy, статика фронтенда)
       │ HTTP (internal)
       ▼
 FastAPI Backend (REST API)
       │                    │
       ▼                    ▼
 PostgreSQL            Redis (очередь задач)
                            │
                            ▼
                      Worker (сканирующий агент)
```

### 3.2 Описание компонентов

| Компонент | Роль | Технология |
|---|---|---|
| **Frontend** | Одностраничное веб-приложение (SPA) | React 18, TypeScript, Vite |
| **Backend API** | REST API, бизнес-логика, аутентификация | Python 3.11, FastAPI, SQLAlchemy (async) |
| **Worker** | Асинхронный обработчик задач сканирования | Python 3.11, asyncio, aiohttp |
| **PostgreSQL** | Основное реляционное хранилище | PostgreSQL 16 |
| **Redis** | Очередь задач, брокер сообщений | Redis 7 |
| **Nginx** | Обратный прокси, раздача статики, TLS | Nginx 1.25 |
| **Certbot** | Получение и обновление TLS-сертификатов | Let's Encrypt / certbot |

### 3.3 Взаимодействие компонентов

1. Клиент обращается к Nginx по HTTPS.
2. Nginx перенаправляет запросы к `/api/*` на Backend, остальные запросы обслуживает статически (SPA).
3. Backend принимает запросы, выполняет аутентификацию, валидацию и бизнес-логику.
4. При создании задачи сканирования Backend публикует сообщение в очередь Redis.
5. Worker читает очередь Redis и выполняет сканирование: краулинг, анализ заголовков, атака.
6. Результаты сканирования сохраняются в PostgreSQL через SQLAlchemy.
7. Клиент опрашивает Backend для получения статуса и результатов.

## 5. Модель данных

### 5.1 Сущности и связи

**User** — учётная запись пользователя:
- `id` (UUID7), `email`, `username`, `hashed_password`
- `role` (user | admin), `is_active`, `totp_secret`
- `avatar_url`, `created_at`, `updated_at`

**UserSession** — активная сессия пользователя:
- `id`, `user_id` (FK → User), `refresh_token_hash`
- `user_agent`, `ip_address`, `created_at`, `expires_at`

**APIToken** — API-токен для программного доступа:
- `id`, `owner_id` (FK → User), `name`
- `token_hash`, `created_at`, `last_used_at`

**Scan** — задача сканирования:
- `id`, `owner_id` (FK → User), `target_url`
- `status` (pending | running | paused | done | cancelled | error)
- `started_at`, `finished_at`

**Vulnerability** — найденная уязвимость:
- `id`, `scan_id` (FK → Scan), `vuln_type`
- `url`, `param`, `payload`, `severity`
- `evidence` (JSON), `created_at`

**Wordlist** — пользовательский словарь пейлоадов:
- `id`, `owner_id` (FK → User), `name`
- `file_path`, `file_size`, `is_builtin`, `created_at`

### 5.2 Диаграмма связей (упрощённая)

```
User ──< UserSession
User ──< APIToken
User ──< Scan ──< Vulnerability
User ──< Wordlist
```

## 6. API

Система предоставляет REST API в соответствии с принципами REST. Документация API доступна по адресу `/api/docs` (Swagger UI) и `/api/redoc`.

### 6.1 Основные группы эндпоинтов

| Группа | Префикс | Назначение |
|---|---|---|
| Аутентификация | `/api/auth` | Регистрация, вход, 2FA, OAuth, токены |
| Пользователи | `/api/users` | Профиль, аватар |
| Сканирования | `/api/scans` | CRUD сканирований, управление |
| Отчёты | `/api/scans/{id}/report` | Результаты и уязвимости |
| Словари | `/api/wordlists` | Загрузка и управление словарями |
| Администрирование | `/api/admin` | Управление пользователями и токенами |
| Служебные | `/health` | Проверка работоспособности |

### 6.2 Аутентификация запросов

Все защищённые эндпоинты требуют заголовка:

```
Authorization: Bearer <access_token>
```

Альтернативно принимается API-токен в том же заголовке.
